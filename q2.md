**Summary**

This document describes a stored Cross-Site Scripting (XSS) vulnerability in the supplied Flask application, shows a step-by-step attack flow that allows a low-privileged user to impersonate an administrator and delete all messages, provides an exact proof-of-concept payload (including the hard-coded admin password) for reproducible testing, and lists remediation steps.

**Affected Components**

- `XSSApp-1.0/XSSApp/app.py`: `add_message()` stores HTML without escaping and marks it with `jinja2.Markup`; `/login` and `/drop_all_messages` are GET endpoints that change server state. 
- `XSSApp-1.0/XSSApp/validation_utilities.py`: `validate_message()` only blocks `<script>` tags and attributes beginning with `on`, but allows other dangerous tags/URLs.
- `XSSApp-1.0/XSSApp/templates/index.html`: message rendering uses `{{ message.message }}` which is rendered as safe HTML because the app stored `Markup`.

**Vulnerability Description**

Type: Stored (persistent) XSS.

Root cause:
- User-supplied `message` is validated insufficiently and then explicitly wrapped with `jinja2.Markup` in `add_message()`, disabling template escaping.
- Admin actions (`/login` and `/drop_all_messages`) are implemented as GET endpoints and rely on session state; therefore a payload that causes the admin's browser to issue those GET requests will execute with the admin's session.

Impact:
- Any attacker who can submit a message can store HTML that executes when an administrator views the messages. Using same-origin requests (iframes), the attacker can trigger admin-only endpoints and perform destructive actions such as clearing all messages.

**Attack Phases**

1) Recon

- Inspect `XSSApp-1.0/XSSApp/app.py` and `templates/index.html` to find where messages are stored and rendered.
- Locate admin endpoints: `/login` (GET with `password` query) and `/drop_all_messages` (GET).
- Find hard-coded password value in `app.py`: `VERY_SECRET_PASSWORD = "55ce7573ece627be0140648f5e403121"` which is accepted by `test_administrator_password()` when hashed.

2) Injection

- Use the contact form on the site and paste a payload into the `Message` textarea. The site will accept and store the payload because validation does not forbid iframes or other dangerous tags.

3) Trigger / Execution

- When an administrator (authenticated via session cookie) loads the page that lists messages, the browser parses and renders the stored HTML payload. If the payload embeds same-origin requests (iframes) pointed at `/login?password=...` and `/drop_all_messages`, the browser will execute those GETs using the admin's cookies, causing the server to mark the session as admin and then clear messages.

4) Cleanup / Persistence

- After `/drop_all_messages` runs successfully, the messages list becomes empty. Because storage is in-memory, this state persists until the app restarts or new messages are added.

**Remediation**

Apply these mitigations to eliminate the attack path:

1. Stop disabling template escaping: remove `jinja2.Markup(result["message"])` in `add_message()` and store raw text. Allow Jinja to auto-escape output, or explicitly escape with `markupsafe.escape()`.
2. Harden `validate_message()` in `XSSApp-1.0/XSSApp/validation_utilities.py`: disallow dangerous tags (`iframe`, `object`, `embed`, `svg`, `math`, `form`, `base`) and block dangerous URL schemes in attributes (`javascript:`, `data:`, `vbscript:`). Prefer a safe HTML sanitizer (e.g., Bleach) with a strict allowlist.
3. Change admin actions to require POST and CSRF protection. Do not implement state-changing operations via GET. Require authentication for `/drop_all_messages` and perform authorization checks server-side.
4. Remove hard-coded secrets from source. Keep secrets out of code and use environment variables or a secure secret store. Rotate the administrative password immediately.
5. Set `SESSION_COOKIE_HTTPONLY = True` and `SESSION_COOKIE_SECURE = True` (when using TLS) to protect session cookies from being read by JS.

**Verification checklist (after fixes)**

- Submit the previous PoC payload and confirm it is sanitized or rejected.
- As admin, visit the messages page and confirm the payload does not perform the login/drop requests.
- Confirm server logs show blocked input or rejected requests.

**Weak code flow (exact vulnerable snippets & file references)**

Below are the concrete weak code locations and the minimal excerpts that create the attack surface. Each snippet is followed by a short note describing why it is dangerous and how it fits into the attack flow.

- `XSSApp-1.0/XSSApp/app.py` — storing unescaped HTML and GET-based admin actions:

```python
# Top secret value! Nobody knows this value and shall never know!
VERY_SECRET_PASSWORD = "55ce7573ece627be0140648f5e403121"

def add_message(args: dict):
   required_args = ["name", "phone_number", "email", "subject", "message"]
   result = {name: args.get(name) for name in required_args}
   for validator_name, validator in VALIDATORS.items():
      if validator_name in result:
         validator(result[validator_name])
   result["message"] = jinja2.Markup(result["message"])
   if is_administrator_logged_in():
      result["name"] = result["name"] + " (Administrator)"

   messages.append(result)

@app.route("/login", methods=["GET"])
def login():
   attempt_admin_login()
   return flask.redirect('/')

@app.route("/drop_all_messages")
def drop_all_messages():
   if is_administrator_logged_in():
      messages.clear()
   return flask.redirect('/')
```

Why this is weak:
- `VERY_SECRET_PASSWORD` is embedded in source; it can be used directly in attack payloads in this assignment context.
- `add_message()` explicitly wraps the user-supplied `message` with `jinja2.Markup`, which tells the templating engine that the content is safe and must not be escaped. This turns otherwise-escaped markup into executable HTML.
- `/login` and `/drop_all_messages` are GET endpoints that change session/server state; same-origin GET requests from embedded content execute with the admin's session.

- `XSSApp-1.0/XSSApp/validation_utilities.py` — weak message validation allowing dangerous tags/URLs:

```python
def validate_message(message: str):
   parsed = BeautifulSoup(message, 'html.parser')
   if parsed.findChildren("script"):
      raise ValueError("message contains scripts!")
   all_attributes = get_all_attributes(parsed)
   if any(has_on_attribute(elem) for elem in all_attributes):
      raise ValueError("An on attribute was detected!")
   return True
```

Why this is weak:
- Validation only rejects `<script>` elements and attributes that start with `on*`. It still allows `iframe`, `object`, `embed`, `svg`, `math`, `form`, and other tags and attributes that can cause effects or load same-origin resources.
- It does not check `href`/`src` schemes (e.g., `javascript:` or `data:`), nor does it sanitize attribute values.

- `XSSApp-1.0/XSSApp/templates/index.html` — stored message rendering point:

```html
<h2>Messages</h2>
{% for message in messages %}
<div style="display: block" class="row alert-secondary">
   <h3>Name: {{message.name}} </h3>
   <h3>Phone: {{message.phone_number}} </h3>
   <h3>Mail: {{message.email}}</h3>
   <h3>Subject: {{message.subject}}</h3>
   <p>Message: {{message.message}}</p>
</div>
{% endfor %}
```

Why this is weak:
- The template uses `{{message.message}}`, which is normally auto-escaped by Jinja2. However, because the app stored a `Markup` object, the value is rendered as raw HTML, so any attacker-supplied HTML executes in viewers' browsers.

**Question 3: Why the `<object>` cookie-theft attempt failed**

I tried to use a standard HTML `<object>` tag to load an attacker-controlled payload and steal cookies, but it failed for browser and application reasons:

1. `<object>` is an embedding element, not an execution primitive.
   - It loads an external document or resource into its own browsing context.
   - Unlike an inline script or a same-origin injected HTML fragment, it does not automatically give attacker code direct access to the parent page's DOM.

2. Cookie theft requires script execution in the right origin context.
   - To read `document.cookie`, attacker code must run as JavaScript inside a page that has access to the target origin.
   - The `<object>` payload does not run arbitrary JavaScript in the parent page, so there is no direct way to call `document.cookie` on the victim page.

3. The embedded document is isolated from the parent document.
   - Even if the object loads HTML, it is treated as a separate document.
   - Same-origin policy prevents a cross-origin object from reading the parent page, and the parent cannot read the embedded document unless the origins match and the browser allows access.

4. Modern browsers restrict dangerous `object`-based tricks.
   - Payloads that try to use `data=javascript:...`, `data:text/html,...`, or similar variants are commonly blocked or do not behave consistently.
   - Many browsers no longer execute script the way older `object`-based XSS examples assumed.

5. In this application, the stronger attack path is HTML injection plus same-origin navigation.
   - The stored payload works when it causes the browser to issue requests to `/login` and `/drop_all_messages` as embedded frames.
   - That works because the requests are normal same-origin navigations, not because the payload can directly steal cookies.

Conclusion:
- The `<object>` approach failed because it did not produce JavaScript execution in the victim page and therefore could not access cookies directly.
- The successful vector in this app is to use stored HTML that triggers same-origin requests, not to rely on `<object>` for cookie extraction.

**Question 4: Three additional security weaknesses in the site**

Below are three concrete weaknesses in the application that improve the attack surface, excluding the BeautifulSoup validator issue.

1. `XSSApp-1.0/XSSApp/app.py` stores attacker-controlled HTML with `jinja2.Markup`.

```python
result["message"] = jinja2.Markup(result["message"])
```

Why this is weak:
- This disables Jinja2 escaping for user input.
- It turns attacker-supplied HTML into trusted HTML, which is exactly what makes stored XSS possible.

2. `XSSApp-1.0/XSSApp/app.py` uses GET endpoints for state-changing admin actions.

```python
@app.route("/login", methods=["GET"])
def login():
   attempt_admin_login()
   return flask.redirect('/')

@app.route("/drop_all_messages")
def drop_all_messages():
   if is_administrator_logged_in():
      messages.clear()
   return flask.redirect('/')
```

Why this is weak:
- Login and message deletion change security state, but they are exposed through GET.
- That makes them easy to trigger from embedded content and makes CSRF-style abuse much easier.
- `/drop_all_messages` should require POST plus CSRF protection and authorization checks.

3. `XSSApp-1.0/XSSApp/app.py` contains a hard-coded administrator secret.

```python
VERY_SECRET_PASSWORD = "55ce7573ece627be0140648f5e403121"
```

Why this is weak:
- Secrets should not be committed in source code.
- Once the value is in the repo, anyone who can read the code can reuse it to authenticate as admin.
- The password should be moved to a secure secret store or environment variable and rotated.

4. `XSSApp-1.0/XSSApp/app.py` weakens cookie protection.

```python
app.config["SESSION_COOKIE_HTTPONLY"] = False
```

Why this is weak:
- HttpOnly cookies are harder for injected scripts to read.
- Disabling this protection makes session theft easier if any script execution happens.
- In a secured deployment, this should be `True`, and `SESSION_COOKIE_SECURE` should also be enabled when using HTTPS.

Short conclusion:
- The biggest code-safety problems are unescaped HTML storage, GET-based admin actions, and the hard-coded admin secret.
- The cookie setting weakness makes impact worse if XSS is achieved.
